## 1. Введение

Zvooq — это учебный музыкальный веб‑сервис для коллекционирования, организации и прослушивания музыки. Проект ориентирован на практику работы с DOM, клиентским хранением состояния, файловой загрузкой, а также построением лёгкого SPA‑слоя поверх многостраничной архитектуры.

Сервис позволяет:
- Загружать аудио‑файлы и обложки.
- Создавать альбомы автоматически при множественной загрузке.
- Управлять плейлистами и избранными треками.
- Слушать музыку через глобальный плеер, не прерывая воспроизведение при навигации.
- Вести статистику, работать с архивом и профилем пользователя.

Документация ниже является полноценным описанием продукта, архитектуры, API, логики клиентской части и практических сценариев.

---

## 2. Цели и назначение продукта

Основные цели проекта:

1. **Обучение практикам веб‑разработки.**
   - Работа с DOM.
   - Работа с асинхронным JS и API.
   - Работа с локальными файлами и формами.

2. **Создание минимально полноценного музыкального сервиса.**
   - Загрузка, хранение, воспроизведение.
   - Создание плейлистов.
   - Избранное и персонализация.

3. **Демонстрация архитектуры SPA на основе статичных страниц.**
   - Подмена `<main>` при навигации.
   - Сохранение состояния плеера.
   - Повторная инициализация страниц через `init*` методы.

---

## 3. Ключевые возможности

- **Загрузка треков**: один файл = трек, несколько файлов = альбом.
- **Генерация длительности**: длительность определяется по метаданным аудио.
- **Обложки**: загрузка обложки и отображение в UI.
- **Плейлисты**: создание, добавление/удаление треков.
- **Избранное**: быстрый toggle, синхронизация с пользователем.
- **Архив**: удалённые треки/альбомы хранятся в `archive.json`.
- **Статистика**: агрегаты по жанрам, активности и избранному.
- **Глобальный плеер**: продолжает играть при переходах.
- **Права доступа**: редактировать можно только владельцу или администратору.

---

## 4. Архитектура проекта

Проект сочетает два подхода:

1. **Многостраничное приложение (MPA).**
   - Страницы доступны напрямую: `/index.html`, `/playlist.html`, `/stats.html` и т.д.
2. **SPA‑поведение поверх MPA.**
   - Скрипт `js/spa.js` перехватывает клики по ссылкам, делает `fetch`, подменяет `<main>`, а затем вызывает `init*`.

Серверная часть реализована на **Node.js + Express** и отдаёт:
- статические файлы;
- JSON API;
- загруженные аудио/изображения.

Данные хранятся в JSON‑файлах в `data/`.

## 7. Сценарии использования (User Journeys)

### 7.1 Новый пользователь
1. Переходит на `/auth.html`.
2. Регистрируется.
3. Попадает на главную.
4. Загружает трек или альбом.
5. Создаёт плейлист.
6. Добавляет треки в плейлист.

### 7.2 Прослушивание музыки
1. Открывает главную.
2. Нажимает Play на треке или альбоме.
3. Навигирует на другую страницу — плеер продолжает играть.

### 7.3 Избранное
1. На карточке трека нажимает “В избранное”.
2. Открывает страницу Избранного.
3. Воспроизводит только любимые треки.

## 9. API (серверные маршруты)

### 9.1 Треки

- `GET /api/tracks` — получить все треки.
- `POST /api/tracks` — загрузить трек или альбом (multipart).
- `PUT /api/tracks/:id` — обновить трек.
- `DELETE /api/tracks/:id` — удалить трек в архив.
- `POST /api/tracks/:id/favorite` — toggle избранного.

### 9.2 Альбомы

- `GET /api/albums` — получить все альбомы.
- `PUT /api/albums/:id` — обновить альбом.
- `DELETE /api/albums/:id` — удалить альбом (переместить в архив).

### 9.3 Плейлисты

- `GET /api/playlists`
- `POST /api/playlists`
- `PUT /api/playlists/:id`
- `DELETE /api/playlists/:id`
- `POST /api/playlists/:id/tracks`
- `DELETE /api/playlists/:id/tracks/:trackId`

### 9.4 Пользователи

- `GET /api/users`
- `POST /api/auth/register`
- `POST /api/auth/login`

## 10. Клиентская часть: маршрутизация и SPA

Файл: `js/spa.js`

Ключевая идея:
- Перехват кликов по ссылкам.
- `fetch()` страницы.
- Подмена `<main>` в DOM.
- Вызов `init*` у соответствующей страницы.

Пример фрагмента:

```js
const spaPaths = new Set([
  '/index.html',
  '/playlist.html',
  '/favorite.html',
  '/stats.html',
  '/archive.html',
  '/about.html',
  '/profile.html'
]);
```

Каждая страница предоставляет `initX()` и регистрируется в `window.PageInits`.

## 11. Глобальный плеер

Файл: `js/shared.js`

Плеер создаётся один раз и живёт во всём приложении.
- Состояние сохраняется в `sessionStorage`.
- При навигации воспроизведение не прерывается.

Ключевые функции:
- `Player.init()` — инициализация.
- `Player.play(track, queue)` — воспроизведение.
- `Player.stop(clearQueue)` — остановка при логауте.

Пример:

```js
Player.play(track, tracks);
```

## 33. Расширенная спецификация API

### 33.1 `GET /api/tracks`

**Описание:** вернуть список всех треков.

**Ответ:**

```json
[
  {
    "id": 1773978595588,
    "title": "Название",
    "artist": "Артист",
    "genre": "Genre",
    "duration": 243,
    "audio": "/uploads/audio/file.mp3",
    "cover": "/uploads/images/cover.jpg"
  }
]
```

### 33.2 `POST /api/tracks`

**Описание:** загрузка трека или альбома. Если передан 1 аудиофайл — создаётся трек, если 2+ — альбом.

**Формат:** `multipart/form-data`.

**Поля:**
- `audio` — один или несколько файлов.
- `cover` — файл обложки.
- `title` или `albumTitle`.
- `artist`, `genre`, `description`, `label`, `releaseDate`.
- `durations` (для альбома) или `duration` (для трека).
- `trackTitles`, `trackArtists`, `trackGenres`, `trackDescriptions` (для альбома).
- `userId`.

**Пример ответа (трек):**

```json
{ "success": true, "type": "track", "data": { "id": 1, "title": "Demo" } }
```

**Пример ответа (альбом):**

```json
{ "success": true, "type": "album", "data": { "id": 10, "title": "Album" }, "tracks": [] }
```

### 33.3 `POST /api/tracks/:id/favorite`

**Описание:** toggle избранного.

**Тело:**

```json
{ "userId": 1773978733499 }
```

**Ответ:**

```json
{ "favorited": true, "track": { "id": 1773978595588 } }
```

### 33.4 `GET /api/playlists`

**Описание:** список плейлистов.

### 33.5 `POST /api/playlists`

**Описание:** создать плейлист.

**Тело:**

```json
{ "title": "Мой плейлист", "userId": 1773978733499 }
```

### 33.6 `DELETE /api/playlists/:id?userId=...`

**Описание:** удалить плейлист, доступ только владельцу или админу.

## 37. Файловая загрузка и обработка медиа

Пайплайн загрузки:
1. Пользователь выбирает файлы.
2. Клиент читает длительность через `Audio`.
3. Формируется `FormData`.
4. Сервер сохраняет файлы в `uploads/`.
5. В JSON‑файлы записываются пути.

Пример:

```js
const formData = new FormData();
formData.append('audio', file);
await API.postForm('/api/tracks', formData);
```

## 38. Подсчёт длительности

Функция `getAudioDuration(file)` создаёт временный `Audio`‑элемент и читает `duration` при событии `loadedmetadata`.

Это позволяет:
- Заполнять `duration` в `tracks.json`.
- Суммировать продолжительность альбомов.

## 46. Подробно: Frontend

### 46.1 Общая архитектура фронтенда

Frontend построен как многостраничное приложение с SPA‑оболочкой. Каждый HTML‑файл представляет самостоятельную страницу, но скрипт `js/spa.js` перехватывает навигацию и подменяет только `<main>`, сохраняя общий контекст (например, плеер).

Ключевые принципы:
- HTML‑страницы остаются доступными напрямую.
- При навигации через ссылки используется `fetch` и подмена `main`.
- Для каждой страницы существует `initX()` и регистрация в `window.PageInits`.
- Состояние плеера живёт в `sessionStorage`.

### 46.2 Основные файлы фронтенда

#### `index.html`
Главная страница: отображение треков, альбомов и плейлистов, фильтры, поиск, переходы.

#### `playlist.html`
Страница плейлистов: список, детальная страница плейлиста, модальные окна добавления треков.

#### `favorite.html`
Страница избранного: фильтрация треков по `favoriteTracks`.

#### `stats.html`
Статистика: вывод KPI, топов, активности.

#### `archive.html`
Архив: список удалённых элементов.

#### `auth.html`
Авторизация: вход и регистрация.

#### `add-track.html`
Добавление трека или альбома с динамическими формами.

#### `edit-track.html`
Редактирование трека или альбома.

#### `profile.html`
Профиль пользователя, список избранного и плейлистов.

### 46.3 SPA‑слой (`js/spa.js`)

Функциональность:
- Перехват кликов по ссылкам.
- Получение HTML через `fetch`.
- Замена `<main>`.
- Вызов `init*` для страницы.

Основные этапы:
1. Нажатие на ссылку.
2. Проверка, что путь является SPA‑страницей.
3. Загрузка HTML.
4. Подмена `<main>`.
5. Загрузка JS‑файла страницы.
6. Вызов `initX()`.

### 46.4 Общие утилиты (`js/shared.js`)

Содержит:
- `API.*` — обёртки над `fetch`.
- `Auth.*` — хранение и получение пользователя.
- `showToast` — уведомления.
- `formatDuration`, `formatDate` — форматирование.
- `Player` — глобальный плеер.
- `renderHeader` — отрисовка шапки.
- `ZLog/ZError` — логирование.

### 46.5 Логика главной страницы (`js/index.js`)

Функции:
- `initIndex()` — точка входа страницы.
- `loadData()` — загрузка треков/альбомов/плейлистов.
- `renderAll()` — основной рендеринг.
- `handleCardClick()` — действия с карточками.
- `createPlaylist()` — создание плейлиста.

### 46.6 Логика плейлистов (`js/playlist.js`)

Функции:
- `initPlaylist()` — инициализация.
- `renderPlaylistList()` — список плейлистов.
- `openPlaylistDetail()` — открытие детальной страницы.
- `addTrackToPlaylist()` — добавить трек.
- `removeTrackFromPlaylist()` — удалить трек.

Присутствует защита от повторной загрузки скрипта:

```js
if (window.__playlistScriptLoaded) return;
```

### 46.7 Логика добавления трека (`js/add-track.js`)

Этапы:
1. Выбор аудио.
2. Определение, альбом это или трек.
3. Динамическое построение формы.
4. Подсчёт длительности треков.
5. Формирование `FormData` и отправка на сервер.

Пример:

```js
const durations = await ensureDurations(selectedFiles);
formData.append('duration', durations[0]);
```

### 46.8 Логика избранного (`js/favorite.js`)

- Загружает все треки.
- Фильтрует по `favoriteTracks`.
- Рендерит карточки треков.

### 46.9 Плеер (`Player`)

Ключевые особенности:
- Очередь треков.
- Shuffle, Repeat.
- Прогресс и громкость.
- Восстановление состояния после навигации.

### 46.10 Права доступа на фронтенде

Фронтенд скрывает кнопки редактирования у тех, кто не владелец.
Проверка:

```js
const canEdit = user && (item.ownerId == user.id || user.isAdmin);
```

### 46.11 Логирование

Практически все ключевые действия выводятся в консоль через `ZLog`.

## 47. Подробно: Backend

### 47.1 Архитектура сервера

Backend построен на Express и работает как:
- API сервер.
- статический файловый сервер.
- хранилище JSON‑данных.

### 47.2 Файлы данных

- `tracks.json`
- `albums.json`
- `playlists.json`
- `users.json`
- `archive.json`

### 47.3 Загрузка файлов (Multer)

Конфиг:
- `audioStorage` — аудио в `/uploads/audio`.
- `imageStorage` — обложки в `/uploads/images`.

### 47.4 Роуты треков

`POST /api/tracks`:
- Если 1 файл — создаётся трек.
- Если 2+ — создаётся альбом и треки.

`DELETE /api/tracks/:id`:
- Перемещает трек в архив.

### 47.5 Роуты альбомов

`DELETE /api/albums/:id`:
- Удаляет альбом и его треки.
- Добавляет в архив.

### 47.6 Роуты плейлистов

`POST /api/playlists`:
- Проверяет `userId`.
- Создаёт плейлист.

`POST /api/playlists/:id/tracks`:
- Проверяет права.
- Добавляет трек.

`DELETE /api/playlists/:id`:
- Проверяет права.
- Удаляет плейлист.

### 47.7 Авторизация

Авторизация упрощённая:
- Логин/пароль проверяется по `users.json`.
- Нет JWT или сессий.

### 47.8 Админ‑роль

Админ определяется в `users.json` через поле:

```json
"isAdmin": 1
```

### 47.9 Обработка ошибок

Если объект не найден — `404`.
Если нет прав — `403`.
Если ошибка валидации — `400`.

### 47.10 Пример серверного кода

```js
app.post('/api/playlists/:id/tracks', (req, res) => {
  const pl = playlists.find(p => p.id == req.params.id);
  if (!pl) return res.status(404).json({ error: 'Not found' });
  if (!canEditPlaylist(pl, req.body.userId)) return res.status(403).json({ error: 'Forbidden' });
  pl.tracks.push(req.body.trackId);
  res.json(pl);
});
```

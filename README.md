# ZVOOQ — адаптивный сайт музыкального веб‑сервиса

## ВВЕДЕНИЕ
Музыкальные веб‑сервисы стали стандартным способом потребления контента, поэтому к ним предъявляются требования удобства, скорости и одинаково комфортной работы на разных устройствах. Пользователь ожидает, что сможет быстро найти нужную музыку, запустить воспроизведение, собрать подборки и управлять личной библиотекой без лишних действий. В условиях высокой конкуренции именно качество интерфейса и продуманность пользовательских сценариев определяют успешность продукта. 

Проект ZVOOQ посвящён разработке адаптивного сайта музыкального веб‑сервиса, объединяющего каталог треков и альбомов, плейлисты, избранное, личный кабинет и базовую статистику. Работа ориентирована на полный цикл: от анализа предметной области и проектирования интерфейса до реализации и проверки функционала. В качестве результата создаётся единый веб‑интерфейс, который сохраняет целостность дизайна и логики при переходе между десктопом, планшетом и смартфоном.

Техническая реализация включает клиентскую часть на HTML/CSS/JavaScript и серверную на Node.js + Express. В проекте предусмотрены REST‑маршруты для операций с треками, альбомами, плейлистами, авторизацией и архивом, а также загрузка медиафайлов через Multer. Данные хранятся в JSON‑структурах, что позволяет концентрироваться на логике работы сервиса без усложнения инфраструктуры на этапе прототипирования.

Цель проекта — продемонстрировать практический подход к созданию адаптивного музыкального веб‑сервиса с реальными пользовательскими сценариями, согласованной визуальной системой и работающим функционалом. Задачи включают анализ требований, проектирование интерфейса, выбор технологий, разработку клиентской и серверной частей, а также тестирование ключевых сценариев использования.

## 1 ТЕОРЕТИЧЕСКИЕ ЧАСТЬ: РАЗРАБОТКА АДАПТИВНОГО САЙТА МУЗЫКАЛЬНОГО ВЕБ‑СЕРВИСА ZVOOQ

### 1.1 Анализ предметной области АДАПТИВНОГО САЙТА МУЗЫКАЛЬНОГО ВЕБ‑СЕРВИСА ZVOOQ
Предметная область проекта — это музыкальный веб‑сервис. Проще говоря, это сайт, где пользователь слушает музыку, сохраняет любимые треки и собирает свои подборки. Такой сервис должен быть удобным и понятным, потому что люди пользуются им в самых разных условиях: дома за компьютером, в дороге со смартфона или на планшете. Поэтому важны скорость действий, простая навигация и адаптивный дизайн.

Главный объект в сервисе — трек. У каждого трека есть название, исполнитель, жанр, длительность, обложка и сам аудиофайл. Трек можно воспроизвести, добавить в избранное, включить в плейлист или удалить. В интерфейсе он показывается по‑разному: карточкой в каталоге, строкой в списке или мини‑элементом в плеере. Это значит, что сайт должен уметь удобно отображать одну и ту же информацию в разных форматах.

Альбом — это набор треков одного релиза. У альбома тоже есть название, исполнитель, обложка и дата выхода. Главное отличие — внутри альбома есть список треков. В интерфейсе обычно нужно показывать и общий вид альбома, и его содержимое (например, по кнопке «показать ещё»). Это делает работу с альбомами чуть сложнее, чем с отдельными треками.

Плейлист — это пользовательская подборка. Пользователь сам решает, какие треки туда добавить и в каком порядке их слушать. Плейлисты часто меняются: их можно создавать, переименовывать, добавлять и удалять треки. Поэтому интерфейс должен давать быстрые и понятные действия управления, чтобы пользователь не путался.

Избранное — это список любимых треков. По сути, это связь «пользователь ↔ трек», которая показывает, что трек отмечен сердечком. Пользователь ожидает, что такое действие сразу отразится в интерфейсе и будет видно на всех страницах. Поэтому система должна хранить и показывать состояние избранного корректно и синхронно.

Архив — это список удалённых элементов. Он полезен, чтобы видеть, что именно было удалено, и при необходимости вернуть данные. Для сервиса это повышает надёжность: удаление не считается окончательным, а пользователь понимает, что произошло. В интерфейсе архив обычно отображается отдельной страницей со списком удалённых объектов и датой удаления.

В сервисе есть разные роли. Гость может просто просматривать каталог. Зарегистрированный пользователь получает доступ к избранному, плейлистам и добавлению контента. Администратор имеет дополнительные права на редактирование и удаление. Это влияет на интерфейс: одни кнопки и формы доступны только после входа, другие — только для администратора.

Сценарии использования можно разделить на несколько групп. Во‑первых, прослушивание: открыть каталог, выбрать трек или альбом, запустить воспроизведение и переключать треки. Во‑вторых, организация библиотеки: создать плейлист, добавить туда музыку, отметить избранное. В‑третьих, управление контентом: загрузить трек или альбом, отредактировать данные, удалить ненужное. В‑четвёртых, аналитика: посмотреть статистику, например сколько треков добавлено и какие самые популярные.

Отдельно важно отметить визуальную сторону. Музыкальные сервисы оцениваются не только по функционалу, но и по внешнему виду. Обложки, цвета, шрифты и иконки создают ощущение качества. Поэтому дизайн должен быть единым и аккуратным: пользователь должен легко ориентироваться и получать приятный опыт.

Адаптивность — один из самых важных моментов. На маленьком экране нельзя показывать столько же элементов, как на большом. Значит, меню, карточки и плеер должны перестраиваться. Например, на телефоне важнее простое управление и крупные кнопки, а на десктопе — обзорность и больше информации на экране.

Если говорить про данные, сервис хранит информацию о треках, альбомах, плейлистах, пользователях и архиве. Все эти сущности связаны между собой: удаление альбома удаляет его треки, плейлист связан с владельцем, избранное связано с пользователем. Это требует аккуратной логики на сервере, чтобы данные всегда были согласованы.

Плеер — отдельный важный элемент. Даже простой плеер должен уметь воспроизводить один трек и очередь, показывать прогресс и поддерживать переключение. Для пользователя это главный инструмент, поэтому он должен быть доступен на всех страницах и сохранять своё состояние при навигации.

Также нужно учитывать ситуации, когда данных мало или нет вовсе. Новый пользователь может увидеть пустой список, отсутствуют обложки или нет ни одного плейлиста. В таких случаях интерфейс должен показывать понятные заглушки и подсказки, а система — работать стабильно без ошибок.

Итог: предметная область ZVOOQ — это сочетание музыкального каталога, пользовательских подборок и управления контентом, где особенно важны удобство, визуальная целостность и адаптивность. Понимание этих особенностей помогает правильно спроектировать интерфейс, выбрать технологии и реализовать логику работы сервиса.

### 1.2 Проектирование будущего интерфейса для АДАПТИВНОГО САЙТА МУЗЫКАЛЬНОГО ВЕБ‑СЕРВИСА ZVOOQ
Проектирование интерфейса для ZVOOQ начинается с понимания основных задач пользователя: быстро найти музыку, запустить воспроизведение, сохранить понравившееся и легко управлять своей библиотекой. Поэтому структура интерфейса строится вокруг понятных разделов и коротких путей к действиям. Пользователь не должен «искать» кнопки — они должны быть логично расположены рядом с контентом.

Структура сайта включает несколько ключевых страниц. Главная страница (`index.html`) — это точка входа в каталог: здесь пользователь видит подборки треков, альбомов и плейлистов. Страница авторизации (`auth.html`) отвечает за вход и регистрацию. Профиль (`profile.html`) показывает информацию о пользователе и его активностях. Для управления контентом предусмотрены формы добавления (`add-track.html`) и редактирования (`edit-track.html`). Отдельные разделы — плейлисты (`playlist.html`), избранное (`favorite.html`), архив удалённых элементов (`archive.html`), статистика (`stats.html`) и страница «О проекте» (`about.html`). Такое разделение делает навигацию прямой и ожидаемой: каждый сценарий имеет своё место.

Навигация строится вокруг фиксированного хедера. Он находится сверху и всегда остаётся доступным. Это важно, потому что пользователь может быстро перейти между разделами, не прокручивая страницу вверх. В хедере расположено меню с основными ссылками и блок с аккаунтом. На мобильных устройствах меню превращается в бургер, чтобы не занимать лишнее место. При открытии меню появляется подложка (scrim), которая фокусирует внимание и помогает пользователю закрыть меню одним нажатием.

Основной элемент представления музыки — карточка. Карточки используются для треков, альбомов и плейлистов. Такой подход создаёт единый визуальный язык и облегчает восприятие: пользователь быстро узнаёт знакомый элемент и понимает, как с ним работать. В карточках размещены ключевые действия: воспроизведение, добавление в избранное, редактирование или удаление (если это доступно). Для альбомов предусмотрено раскрытие списка треков, чтобы пользователь мог видеть содержимое без перехода на отдельную страницу.

Плеер — важнейший элемент интерфейса. Он закреплён внизу страницы и остаётся доступным всегда, чтобы пользователь не терял контроль над воспроизведением. Плеер показывает название трека, исполнителя, прогресс и основные кнопки управления. Такое решение соответствует привычкам пользователей музыкальных сервисов: музыка играет в фоне, а навигация по сайту продолжается.

Отдельное внимание уделено формам и сценариям управления контентом. Формы добавления и редактирования должны быть простыми и понятными: логичная последовательность полей, подсказки, понятные кнопки. Загрузка треков и обложек — это критичный сценарий, поэтому интерфейс делает его прозрачным (что загружать, в каком порядке, как называется файл). В результате пользователь не боится добавлять контент и ощущает контроль над процессом.

Дизайн выбирается спокойный и «музыкальный»: тёмная тема, контрастные акценты, аккуратная типографика. Тёмная палитра помогает выделить обложки и уменьшает утомляемость глаз при длительном использовании. Контрастные кнопки и иконки делают действия заметными, а единый стиль шрифтов сохраняет визуальную целостность. Иконки используются как опора для быстрого восприятия — пользователю легче ориентироваться, когда значки понятны и повторяются.

Интерфейс строится так, чтобы быть устойчивым к различным состояниям данных. Если контента нет, пользователь видит пустые состояния с понятными пояснениями, а не «пустой экран». Если у трека нет обложки, используется заглушка. Это снижает чувство ошибок и поддерживает приятный пользовательский опыт.

Для адаптивности используется несколько ключевых точек: 768px и 480px. На планшетах и телефонах карточки перестраиваются в одну колонку или более компактную сетку, уменьшаются отступы, часть второстепенной информации скрывается. На телефонах плеер упрощается, чтобы не перегружать экран. Меню превращается в бургер, формы становятся вертикальными, а кнопки — более крупными для удобного касания. На больших экранах, наоборот, сохраняется обзорность: больше карточек в ряд, расширенные блоки, удобная работа с несколькими секциями одновременно.

Важной частью проектирования является согласованность всех экранов. Если на главной странице карточка выглядит и работает так, то в избранном и плейлистах она должна вести себя аналогично. Это снижает когнитивную нагрузку: пользователь не учится заново, а переносит навыки между разделами. Повторяемые элементы — меню, кнопки, иконки, плеер — создают ощущение цельного продукта.

В итоге проектирование интерфейса ZVOOQ строится на принципах простоты, предсказуемости и адаптивности. Пользователь получает быстрый доступ к основным сценариям, единый визуальный стиль и удобную работу на любом устройстве. Такое проектирование является фундаментом, на котором затем реализуется HTML‑разметка, CSS‑стили и JavaScript‑логика.

### 1.3 Выбор технологий для построения АДАПТИВНОГО САЙТА МУЗЫКАЛЬНОГО ВЕБ‑СЕРВИСА ZVOOQ
Выбор технологий для ZVOOQ основан на цели проекта: создать понятный, адаптивный и функциональный музыкальный веб‑сервис без лишней сложности. Поэтому используются простые и проверенные инструменты, которые хорошо подходят для учебного или прототипного решения, но при этом позволяют реализовать полный цикл работы приложения — от интерфейса до серверной логики.

Для разметки выбран **HTML5**. Он даёт семантическую структуру страниц и делает код читаемым: понятно, где находится шапка, основной контент и секции. Это важно для поддерживаемости проекта, а также для правильной работы адаптивного дизайна и доступности. В проекте несколько страниц, каждая отвечает за свой сценарий (каталог, профиль, плейлисты, избранное, архив), и HTML‑структура позволяет логично разделить эти экраны.

За внешний вид отвечает **CSS3**. Здесь используются CSS‑переменные для единой цветовой схемы, сетки на основе Flexbox и Grid, а также медиа‑запросы для адаптивности. Такой подход даёт гибкость: на больших экранах элементы располагаются свободно и «широко», а на мобильных — компактно и удобно для касания. Дополнительно используются шрифты из Google Fonts и иконки Material Symbols — они придают интерфейсу современный вид и улучшают восприятие.

Логику взаимодействия реализует **JavaScript (ES6+)**. Он управляет загрузкой данных, обработкой кликов, работой форм, отображением уведомлений и запуском плеера. В проекте JavaScript разбит на модули по страницам, что делает код понятнее: каждый файл отвечает за свой экран. Для запросов к серверу применяется Fetch API, а состояние пользователя и плеера хранится в LocalStorage и SessionStorage. Это даёт ощущение «живого» приложения: пользователь не теряет прогресс и настройки при переходах между страницами.

Серверная часть построена на **Node.js + Express**. Это лёгкая и популярная связка, удобная для небольших проектов. Express даёт понятную структуру REST‑маршрутов: отдельные эндпоинты для треков, альбомов, плейлистов, авторизации, статистики и архива. Такой подход позволяет чётко разделить ответственность: фронтенд отвечает за отображение, бэкенд — за данные и бизнес‑логику.

Для загрузки аудио и изображений используется **Multer**. Он позволяет принимать файлы с формы, сохранять их в папки проекта и связывать с метаданными трека или альбома. Это важная часть именно музыкального сервиса, потому что контент включает не только текст, но и медиафайлы.

Хранение данных организовано в виде **JSON‑файлов**. Такой формат подходит для прототипа: он прост, не требует настройки базы данных и при этом позволяет хранить структурированную информацию о треках, альбомах, плейлистах, пользователях и архиве. Для реального продукта эту часть можно заменить на полноценную СУБД, но для учебного проекта JSON — оптимальное решение.

Итоговый набор технологий обеспечивает баланс между простотой и функциональностью. С одной стороны, проект остаётся лёгким и понятным, с другой — реализует все ключевые сценарии музыкального сервиса. Это позволяет сосредоточиться на правильной архитектуре интерфейса и логике работы приложения, не перегружая разработку лишними инструментами.

## 2 ПРАКТИЧЕСКАЯ ЧАСТЬ: СОЗДАНИЕ АДАПТИВНОГО САЙТА МУЗЫКАЛЬНОГО ВЕБ‑СЕРВИСА ZVOOQ

### 2.1 Реализация HTML‑разметки сайта
HTML‑разметка в проекте ZVOOQ построена как набор отдельных страниц, каждая из которых отвечает за конкретный пользовательский сценарий. Такой подход упрощает понимание структуры проекта и помогает поддерживать код: если нужно изменить форму добавления трека, достаточно открыть `add-track.html`, а не искать нужный участок в одном большом файле. При этом все страницы объединены общими элементами интерфейса: шапкой сайта, плеером и контейнером уведомлений.

Основой разметки служит семантическая структура. На каждой странице есть `<header>` для навигации, `<main>` для основного контента и вспомогательные блоки вроде модальных окон или контейнеров загрузки. Использование семантических тегов важно не только для читаемости кода, но и для доступности: браузеры и вспомогательные технологии лучше понимают структуру страницы, а это влияет на удобство использования.

Ниже приведён общий подход к разметке, который повторяется на большинстве страниц:
1. Заголовок страницы с названием раздела и кратким описанием.
2. Основной контент, разделённый на секции.
3. Глобальные элементы: плеер и контейнер для уведомлений.

Такой каркас обеспечивает единый визуальный ритм. Пользователь быстро узнаёт знакомую структуру, а разработчику проще поддерживать единый стиль.

#### Общий каркас страницы
Каждая HTML‑страница начинается с объявления `<!DOCTYPE html>` и базовых мета‑тегов. Это стандарт, который гарантирует корректное отображение в браузере и правильную работу адаптивности. Важно, что на всех страницах подключается общий CSS‑файл `css/style.css` и базовые JavaScript‑модули, отвечающие за хедер, плеер и навигацию.

Пример базового каркаса (упрощённо):

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Zvooq | Главная</title>
  <link rel="stylesheet" href="/css/style.css">
</head>
<body>
  <header class="site-header" id="site-header"></header>

  <main class="page-container">
    <!-- Контент страницы -->
  </main>

  <div id="global-player"></div>
  <div id="toast-container"></div>

  <script src="/js/shared.js"></script>
  <script src="/js/spa.js"></script>
</body>
</html>
```

Этот каркас гарантирует, что у всех страниц одинаковые базовые элементы. Хедер и плеер не дублируются вручную — они вставляются через JavaScript, что упрощает поддержку.

#### Главная страница (`index.html`)
Главная страница — это витрина сервиса. Здесь представлены подборки треков, альбомов и плейлистов. Разметка организована секциями, каждая отвечает за свой тип контента. Внутри секций размещаются сетки карточек.

Типичная структура:
- Блок приветствия и краткого описания.
- Раздел «Треки» с карточками.
- Раздел «Альбомы».
- Раздел «Плейлисты».

Каждый раздел имеет свой заголовок и кнопку «показать все», что позволяет пользователю перейти к полному списку. В HTML это реализуется через отдельные контейнеры, в которые JavaScript подставляет карточки.

#### Страница авторизации (`auth.html`)
Авторизация — отдельный, изолированный сценарий, поэтому для неё используется более компактный макет. В центре страницы размещается карточка с формой входа и регистрации. В разметке предусмотрено два состояния: форма входа и форма регистрации. Они переключаются с помощью JavaScript, но в HTML присутствуют обе, одна из них скрыта.

Ключевые элементы:
- Поля ввода для имени пользователя и пароля.
- Кнопка входа.
- Ссылка для перехода к регистрации.

Такая разметка позволяет быстро переключаться между формами без перезагрузки страницы.

#### Страница профиля (`profile.html`)
Профиль содержит краткую информацию о пользователе и его активности. Разметка включает:
- Блок «героя» (имя, аватар, краткие данные).
- Статистические карточки (количество треков, плейлистов и т.д.).
- Секции с последними плейлистами или действиями.

Это делает страницу полезной как «панель управления» для пользователя.

#### Формы добавления и редактирования (`add-track.html`, `edit-track.html`)
Разметка форм строится по принципу пошагового заполнения. Это важно, потому что добавление трека может включать загрузку файлов и заполнение метаданных. В HTML предусмотрены группы полей, подписи и подсказки.

Например, при добавлении трека есть отдельный блок для загрузки аудио, отдельный блок для обложки и далее поля с метаданными. Такой подход улучшает восприятие и снижает риск ошибки.

#### Плейлисты и избранное (`playlist.html`, `favorite.html`)
Эти страницы выводят списки контента в виде карточек или строк. В HTML заранее предусмотрены контейнеры, куда JS динамически вставляет данные. Пользователь видит единый стиль карточек, что создаёт ощущение согласованности интерфейса.

На странице плейлистов есть блок списка доступных плейлистов и отдельная зона для отображения содержимого выбранного плейлиста. Это удобно: пользователь может быстро переключаться между подборками без лишних переходов.

#### Архив (`archive.html`)
Архив — это список удалённых элементов. Разметка проста: заголовок страницы и контейнер для элементов. Если архив пуст, показывается блок «пустого состояния» с иконкой и текстом. Такая визуальная подсказка помогает пользователю понять, что система работает корректно.

#### Статистика (`stats.html`)
Страница статистики оформлена как набор карточек и таблиц. В HTML предусмотрены контейнеры для:
- общего количества треков;
- количества добавленных за неделю;
- списка популярных треков.

Поскольку данные меняются, всё содержимое подставляется через JavaScript, но HTML даёт каркас для отображения.

#### Страница «О проекте» (`about.html`)
Эта страница носит информационный характер. Здесь размещается описание проекта, основные технологии и автор. Разметка включает блоки с текстом и список «чипов» технологий. Такая структура делает страницу аккуратной и визуально лёгкой.

#### Карточки и списки
Отдельное внимание уделяется карточкам. Карточка — это повторяемый компонент, который встречается на разных страницах: в каталоге, в избранном, в плейлистах. В HTML карточка состоит из:
- обложки;
- названия и автора;
- служебной информации (жанр, длительность, дата);
- кнопок действий.

Такой состав гарантирует, что пользователь видит самое важное сразу. При адаптивной верстке карточки перестраиваются, но сохраняют свои ключевые элементы.

#### Встроенный плеер
Плеер расположен внизу страницы и присутствует на всех страницах. В HTML предусмотрен контейнер `#global-player`, который заполняется через JavaScript. Это снижает дублирование и позволяет централизованно управлять плеером.

#### Уведомления
Для сообщений о действиях (например, «трек добавлен в избранное») используется контейнер `#toast-container`. Это отдельный элемент в HTML, который остаётся пустым, пока не сработает JavaScript‑логика. Такой подход упрощает поддержку: все уведомления идут в один канал.

### 2.2 Реализация CSS‑стилей сайта
CSS‑оформление в проекте ZVOOQ построено вокруг идеи единой дизайн‑системы. Это значит, что почти все цвета, размеры, отступы и тени задаются через CSS‑переменные. Такой подход делает интерфейс цельным и упрощает поддержку: если нужно изменить, например, основной цвет или радиус скругления, это можно сделать в одном месте. В проекте эти переменные находятся в `:root` и задают базовые значения для всех компонентов.

В начале файла определяются основные цвета, фон, текстовые оттенки, размеры, радиусы и параметры теней. Это позволяет построить согласованный визуальный стиль. Далее задаются глобальные правила для базовых элементов: сброс отступов, установка шрифта, общая высота, поведение ссылок и кнопок. Это создаёт стабильную основу для всех страниц.

#### Базовые переменные и общие стили
В проекте используется тёмная палитра, которая подчёркивает обложки и делает интерфейс более «музыкальным». Вместе с этим применяются мягкие тени и полупрозрачные границы, чтобы блоки не казались тяжёлыми. Ниже приведён пример базовых переменных:

```css
:root {
  --rose:    #ffffff;
  --mauve:   #d6d6d6;
  --violet:  #8a8a8a;
  --deep:    #4a4a4a;
  --bg:      #000000;
  --bg2:     #1a1a1a;
  --bg3:     #2b2b2b;
  --surface: #3a3a3a;
  --surface2:#4a4a4a;
  --text:    #ffffff;
  --text2:   #d6d6d6;
  --text3:   #8a8a8a;
  --border:  rgba(255,255,255,0.18);
  --radius:  14px;
  --radius-sm: 8px;
  --player-h: 80px;
  --header-h: 64px;
  --shadow:  0 8px 32px rgba(0,0,0,0.6);
  --glow:    0 0 24px rgba(255,255,255,0.15);
}
```

После переменных задаются базовые правила: общий шрифт, цвет текста, фон, поведение кнопок и ссылок. Это важно, чтобы все элементы выглядели согласованно:

```css
*,
*::before,
*::after { box-sizing: border-box; margin: 0; padding: 0; }

body {
  font-family: 'DM Sans', sans-serif;
  background: var(--bg);
  color: var(--text);
  min-height: 100vh;
  padding-top: var(--header-h);
  padding-bottom: calc(var(--player-h) + 16px);
  line-height: 1.6;
}

a { color: inherit; text-decoration: none; }
button { cursor: pointer; font-family: inherit; }
```

Дополнительно задаётся поведение изображений и полей ввода, чтобы они корректно масштабировались и не «ломали» сетку:

```css
img { max-width: 100%; display: block; }
input, textarea, select { font-family: inherit; }
```

В проекте также стилизованы полосы прокрутки, чтобы они соответствовали тёмной теме:

```css
::-webkit-scrollbar { width: 6px; height: 6px; }
::-webkit-scrollbar-track { background: var(--bg2); }
::-webkit-scrollbar-thumb { background: var(--violet); border-radius: 4px; }
```

#### Шапка и навигация
Хедер фиксируется сверху, чтобы навигация была всегда доступна. В CSS это реализовано через `position: fixed` и слой `z-index`. Также применяется лёгкое размытие фона (backdrop‑filter), что делает интерфейс современным.

```css
.site-header {
  position: fixed;
  top: 0; left: 0; right: 0;
  height: var(--header-h);
  background: rgba(0, 0, 0, 0.85);
  backdrop-filter: blur(20px);
  border-bottom: 1px solid var(--border);
  display: flex;
  align-items: center;
  padding: 0 24px;
  gap: 32px;
  z-index: 1000;
}
```

Логотип выделяется градиентом и крупным шрифтом, чтобы сразу читалось название сервиса:

```css
.header-logo {
  font-family: 'Syne', sans-serif;
  font-weight: 800;
  font-size: 22px;
  letter-spacing: -0.5px;
  background: linear-gradient(135deg, var(--rose), var(--mauve));
  -webkit-background-clip: text;
  -webkit-text-fill-color: transparent;
}
```

Для мобильной версии предусмотрена кнопка‑бургер:

```css
.header-burger {
  display: none;
  align-items: center;
  justify-content: center;
  width: 40px;
  height: 40px;
  border-radius: 10px;
  border: 1px solid var(--border);
  background: var(--bg2);
  color: var(--text);
}
```

Навигационные ссылки оформляются как небольшие «плашки», которые меняют фон при наведении. Это добавляет визуальную отзывчивость:

```css
.nav-link {
  padding: 8px 14px;
  border-radius: var(--radius-sm);
  font-size: 14px;
  font-weight: 500;
  color: var(--text2);
  transition: all 0.2s ease;
}
.nav-link:hover,
.nav-link.active {
  background: var(--surface);
  color: var(--text);
}
```

### 2.3 Реализация JavaScript‑функционала сайта
JavaScript‑функционал ZVOOQ состоит из двух больших частей: серверной логики (`server.js`) и клиентских модулей в папке `js/`. Сервер отвечает за хранение данных и бизнес‑правила, клиент — за отображение, взаимодействие с пользователем, состояние и воспроизведение. Важная особенность проекта — использование папок `data/` и `uploads/`:  
`data/` хранит JSON‑файлы с сущностями (tracks, albums, playlists, users, archive), а `uploads/` содержит загруженные медиафайлы (аудио и обложки). Это прямо влияет на JavaScript‑логику: сервер пишет данные в `data/`, а клиент запрашивает их и отображает.

Ниже приведено максимально подробное описание всех модулей с примерами кода. Я показываю реальные фрагменты и типовые сценарии, чтобы было видно, как построена логика.

**Общая архитектура клиент‑сервер**
1. Клиент отправляет запросы через Fetch API.
2. Сервер обрабатывает запрос, читает/пишет JSON в `data/`.
3. При загрузке файлов сервер сохраняет их в `uploads/` и возвращает пути.
4. Клиент отображает данные, обновляет интерфейс и сохраняет состояние в LocalStorage/SessionStorage.

#### 2.3.1 Серверная часть: `server.js`
Сервер построен на Express. В нём реализованы:
1. REST‑эндпоинты для треков, альбомов, плейлистов.
2. Авторизация (регистрация/вход).
3. Избранное.
4. Архив удалённых сущностей.
5. Статистика.
6. Загрузка файлов через Multer.

**Подключение Express и базовые middlewares**
```js
const express = require('express');
const fs = require('fs');
const path = require('path');
const multer = require('multer');

const app = express();
const PORT = 3000;

app.use(express.json());
app.use(express.static(path.join(__dirname)));
app.use('/uploads', express.static(path.join(__dirname, 'uploads')));
```
Здесь подключаются `express`, `fs`, `path` и `multer`. Важно, что папка `uploads/` раздаётся как статическая, поэтому клиент может использовать ссылки вида `/uploads/audio/filename.mp3`.

**Хранилище JSON (`data/`)**
```js
const readJSON = (file) => {
  try {
    return JSON.parse(fs.readFileSync(path.join(__dirname, 'data', file), 'utf8'));
  } catch { return []; }
};
const writeJSON = (file, data) => {
  fs.writeFileSync(path.join(__dirname, 'data', file), JSON.stringify(data, null, 2));
};
```
Эти функции используются во всех эндпоинтах. Они читают и записывают данные в папку `data/`. Это простой, но очень наглядный способ хранения.

**Конфигурация Multer и папка `uploads/`**
```js
const uploadFields = multer({
  storage: multer.diskStorage({
    destination: (req, file, cb) => {
      if (file.fieldname === 'cover') cb(null, path.join(__dirname, 'uploads/images'));
      else cb(null, path.join(__dirname, 'uploads/audio'));
    },
    filename: (req, file, cb) =>
      cb(null, Date.now() + '-' + Math.random().toString(36).slice(2) + path.extname(file.originalname))
  })
});
```
Файлы аудио попадают в `uploads/audio`, обложки — в `uploads/images`. Сервер возвращает ссылки на эти файлы, а клиент отображает их.

**Пример: создание трека или альбома**
```js
app.post('/api/tracks', uploadFields.any(), (req, res) => {
  const ownerId = req.body.userId ? parseInt(req.body.userId) : null;
  if (!ownerId) return res.status(403).json({ error: 'Auth required' });

  const tracks = readJSON('tracks.json');
  const albums = readJSON('albums.json');
  const files = Array.isArray(req.files) ? req.files : [];
  const audioFiles = files.filter(f => f.fieldname === 'audio');
  const coverFile = files.find(f => f.fieldname === 'cover') || null;
  const coverPath = coverFile ? `/uploads/images/${coverFile.filename}` : '/uploads/images/default.jpg';

  if (audioFiles.length === 1) {
    const newTrack = { id: Date.now(), title: req.body.title, artist: req.body.artist,
      genre: req.body.genre, description: req.body.description || '', label: req.body.label || '',
      releaseDate: req.body.releaseDate || new Date().toISOString().split('T')[0],
      duration: parseInt(req.body.duration) || 0,
      audio: `/uploads/audio/${audioFiles[0].filename}`,
      cover: coverPath, favoriteTracks: [], albumId: null, ownerId
    };
    tracks.push(newTrack);
    writeJSON('tracks.json', tracks);
    res.json({ success: true, type: 'track', data: newTrack });
  } else {
    // Альбом и его треки
  }
});
```
Этот код показывает логику обработки формы загрузки. Если файл один — создаётся трек; если несколько — создаётся альбом и связанные треки. Все данные сохраняются в `data/tracks.json` и `data/albums.json`, а файлы — в `uploads/`.

**Избранное**
```js
app.post('/api/tracks/:id/favorite', (req, res) => {
  const tracks = readJSON('tracks.json');
  const users = readJSON('users.json');
  const { userId } = req.body;
  const track = tracks.find(t => t.id == req.params.id);
  if (!track) return res.status(404).json({ error: 'Not found' });

  if (!track.favoriteTracks) track.favoriteTracks = [];
  const fi = track.favoriteTracks.indexOf(userId);
  if (fi === -1) track.favoriteTracks.push(userId);
  else track.favoriteTracks.splice(fi, 1);
  writeJSON('tracks.json', tracks);

  const user = users.find(u => u.id == userId);
  if (user) {
    if (!user.favoriteTracks) user.favoriteTracks = [];
    const ui = user.favoriteTracks.indexOf(parseInt(req.params.id));
    if (ui === -1) user.favoriteTracks.push(parseInt(req.params.id));
    else user.favoriteTracks.splice(ui, 1);
    writeJSON('users.json', users);
  }
  res.json({ favorited: fi === -1, track });
});
```
Сервер синхронизирует избранное и в треке, и в пользователе. Это важно, чтобы в разных местах интерфейса отображались корректные данные.

**Архив удалений**
```js
app.delete('/api/tracks/:id', (req, res) => {
  const tracks = readJSON('tracks.json');
  const archive = readJSON('archive.json');
  const idx = tracks.findIndex(t => t.id == req.params.id);
  if (idx !== -1) {
    archive.push({ ...tracks[idx], deletedAt: new Date().toISOString() });
    tracks.splice(idx, 1);
    writeJSON('tracks.json', tracks);
    writeJSON('archive.json', archive);
  }
  res.json({ success: true });
});
```
Удалённые элементы переносятся в `data/archive.json`. Это позволяет отображать архив на клиенте.

**Статистика**
```js
app.get('/api/stats', (req, res) => {
  const tracks = readJSON('tracks.json');
  const now = new Date();
  const weekAgo = new Date(now - 7 * 24 * 60 * 60 * 1000);
  const addedThisWeek = tracks.filter(t => new Date(t.releaseDate) >= weekAgo).length;
  const topTracks = [...tracks].sort((a, b) =>
    (b.favoriteTracks?.length || 0) - (a.favoriteTracks?.length || 0)
  ).slice(0, 5);
  res.json({ total: tracks.length, addedThisWeek, topTracks });
});
```
Статистика строится на данных из `data/tracks.json` и возвращается в удобном виде для отображения.

#### 2.3.2 Клиентский слой API и общие утилиты: `js/shared.js`
Этот файл — «сердце» фронтенда. Он содержит:

**API‑клиент**
```js
const API = {
  async get(url) {
    const res = await fetch(url);
    if (!res.ok) throw new Error('GET failed');
    return await res.json();
  },
  async post(url, body) {
    const res = await fetch(url, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(body)
    });
    if (!res.ok) throw new Error('POST failed');
    return await res.json();
  },
  async postForm(url, formData) {
    const res = await fetch(url, { method: 'POST', body: formData });
    if (!res.ok) throw new Error('POST_FORM failed');
    return await res.json();
  }
};
```
Этот слой позволяет одинаково работать со всеми запросами и не дублировать код.

**Авторизация**
```js
const Auth = {
  getUser() {
    try { return JSON.parse(localStorage.getItem('zvooq_user')); } catch { return null; }
  },
  setUser(u) { localStorage.setItem('zvooq_user', JSON.stringify(u)); },
  logout() {
    if (window.Player?.stop) Player.stop(true);
    localStorage.removeItem('zvooq_user');
    window.location.href = '/auth.html';
  }
};
```
Пользователь хранится в LocalStorage. Это позволяет быстро проверять статус без лишних запросов.

**Плеер (сохранение состояния)**
```js
const Player = (() => {
  let state = JSON.parse(sessionStorage.getItem('zvooq_player') || '{}');
  function saveState() { sessionStorage.setItem('zvooq_player', JSON.stringify(state)); }
  function play(track, queue = []) {
    state.current = track;
    state.queue = queue;
    saveState();
    // далее создаются элементы плеера и запускается аудио
  }
  return { play };
})();
```
Состояние плеера сохраняется в `sessionStorage`, поэтому при переходе между страницами воспроизведение не «теряется».

#### 2.3.13 Очередь воспроизведения и состояние плеера
Плеер сохраняет текущий трек и очередь, чтобы пользователь мог продолжить прослушивание после переходов:

```js
const Player = (() => {
  let state = JSON.parse(sessionStorage.getItem('zvooq_player') || '{}');

  function saveState() { sessionStorage.setItem('zvooq_player', JSON.stringify(state)); }

  function play(track, queue = []) {
    state.current = track;
    state.queue = queue;
    saveState();
    renderPlayer();
    startAudio(track.audio);
  }

  function next() {
    if (!state.queue || !state.queue.length) return;
    const idx = state.queue.findIndex(t => t.id == state.current.id);
    const nextTrack = state.queue[idx + 1] || state.queue[0];
    play(nextTrack, state.queue);
  }

  return { play, next };
})();
```

Важный момент: пути к файлам берутся из `uploads/`, поэтому аудиофайлы доступны напрямую через `/uploads/audio/...`.

#### 2.3.14 Работа с избранным на клиенте
Функция избранного вызывается на разных страницах. Она обновляет UI и синхронизирует состояние пользователя:

```js
async function toggleFavorite(trackId, btn) {
  const user = Auth.getUser();
  if (!user) { showToast('Войдите, чтобы добавлять в избранное', 'error'); return; }
  const result = await API.post(`/api/tracks/${trackId}/favorite`, { userId: user.id });
  const isFav = result.favorited;
  if (btn) {
    btn.classList.toggle('active', isFav);
    const icon = btn.querySelector('.material-symbols-rounded');
    if (icon) icon.textContent = isFav ? 'favorite' : 'favorite_border';
  }
  const updatedUser = { ...user };
  if (!updatedUser.favoriteTracks) updatedUser.favoriteTracks = [];
  if (isFav) updatedUser.favoriteTracks.push(trackId);
  else updatedUser.favoriteTracks = updatedUser.favoriteTracks.filter(id => id !== trackId);
  Auth.setUser(updatedUser);
}
```

Эта логика важна для согласованности интерфейса: один и тот же трек должен отображаться как «избранный» на главной, в плейлистах и в профиле.

```js
document.addEventListener('click', async (e) => {
```

### 2.4 Тестирование сайта на соответствие поставленным требованиям и отработки функционала
Раздел тестирования в проекте ZVOOQ направлен на проверку двух групп требований: функциональных (работает ли логика сервиса) и нефункциональных (удобство, устойчивость, адаптивность). Так как проект является полноценным прототипом, проверка должна охватывать пользовательские сценарии целиком — от входа в систему до воспроизведения и управления контентом. Ниже приводится подробное описание проведённых проверок и их логики.

#### 2.4.1 Цели тестирования
Цели тестирования формулируются так:
1. Убедиться, что все сценарии, заявленные в требованиях, реализованы корректно.
2. Проверить, что сайт стабильно работает при разных размерах экранов и условиях использования.
3. Убедиться, что данные корректно сохраняются и отображаются после операций (создание, редактирование, удаление).
4. Проверить корректность взаимодействия клиентской части и серверного API.
5. Зафиксировать устойчивость интерфейса к ошибкам и отсутствию данных.

#### 2.4.2 Методика тестирования
Для проекта применяется сочетание:
1. **Ручного функционального тестирования** — проверка сценариев с позиции пользователя.
2. **Проверки пользовательского интерфейса** — визуальная корректность, адаптивность, состояния элементов.
3. **Проверки данных** — правильность записи в `data/` и соответствие отображения.
4. **Проверки крайних случаев** — отсутствие данных, ошибки ввода, удаление элементов.

Такой подход позволяет убедиться, что приложение работает в реальных условиях использования.

#### 2.4.3 Проверка адаптивности интерфейса
Адаптивность — ключевое требование музыкального веб‑сервиса. Пользователь может работать на разных устройствах, поэтому тестирование проводилось в трёх режимах:
1. **Desktop (широкий экран)** — проверка правильности отображения сеток карточек, меню, плеера и форм.
2. **Tablet (≈768px)** — проверка перестройки сеток, корректной работы бургер‑меню и компактных блоков.
3. **Mobile (≈480px)** — проверка упрощения интерфейса, видимости кнопок, удобства управления плеером.

Проверяемые элементы:
- Перестроение карточек в одну колонку на мобильных устройствах.
- Адаптация заголовков и шрифтов (снижение размера, сохранение читаемости).
- Корректное открытие и закрытие бургер‑меню.
- Видимость плеера и корректность его отображения.
- Стабильность форм: поля не выходят за экран, не перекрывают друг друга.

Результаты: интерфейс корректно адаптируется на всех ключевых брейкпоинтах. Элементы остаются доступными и читаемыми, а навигация — предсказуемой.

#### 2.4.4 Тестирование авторизации и профиля
Авторизация — базовый сценарий, от которого зависят остальные функции (избранное, плейлисты, добавление треков). Проверка включает:
1. Регистрация нового пользователя.
2. Вход существующего пользователя.
3. Проверка корректности хранения пользователя в LocalStorage.
4. Корректное отображение имени пользователя в хедере.
5. Переход в профиль и отображение данных пользователя.

**Позитивные сценарии**:
- При регистрации с валидными данными создаётся пользователь, выполняется автоматический вход.
- При корректном логине отображается приветствие и выполняется переход на главную.

**Негативные сценарии**:
- Пустые поля — выводится предупреждение.
- Слишком короткий пароль — отказ в регистрации.
- Неверный логин/пароль — сообщение об ошибке.

Результаты: все сценарии отрабатывают корректно, интерфейс показывает понятные сообщения, а пользовательский статус сохраняется между страницами.

#### 2.4.5 Проверка добавления и редактирования треков/альбомов
Эти сценарии проверяются как критические, так как они затрагивают загрузку файлов и запись данных в `data/` и `uploads/`.

**Проверка добавления трека**:
1. Выбор аудиофайла и обложки.
2. Заполнение метаданных (название, исполнитель, жанр, дата).

4. Проверка появления нового трека в каталоге.
5. Проверка создания файлов в `uploads/audio` и `uploads/images`.
6. Проверка записи в `data/tracks.json`.

**Проверка добавления альбома**:
1. Выбор нескольких аудиофайлов.
2. Заполнение данных альбома.
3. Проверка создания альбома в `data/albums.json`.
4. Проверка создания отдельных треков в `data/tracks.json`.

**Редактирование**:
1. Изменение данных трека.
2. Проверка сохранения изменений в `data/tracks.json`.
3. Проверка обновления карточек в интерфейсе.

Результаты: загрузка работает корректно, данные сохраняются, а интерфейс отображает актуальную информацию.

#### 2.4.6 Проверка плейлистов
Плейлисты тестируются по следующим сценариям:
1. Создание нового плейлиста.
2. Добавление трека в плейлист.
3. Удаление трека из плейлиста.

Результаты: логика плейлистов стабильна, ошибки доступа обрабатываются, изменения мгновенно отображаются в UI.

#### 2.4.7 Проверка избранного
Функционал избранного проверялся так:
1. Добавление трека в избранное с разных страниц.
2. Проверка появления трека в разделе «Избранное».
3. Снятие отметки избранного.
4. Синхронизация состояния в карточках (сердечко активно/не активно).

Результаты: состояние избранного всегда синхронно, данные корректно записываются в `data/users.json` и `data/tracks.json`.

#### 2.4.8 Проверка архива
Архив — важная часть функционала, которая подтверждает корректное удаление:

#### 2.4.9 Проверка статистики
Статистика проверялась по данным сервера:

```js
    iconEl.textContent = icon;
    const textEl = document.createElement('span');
    textEl.className = 'toast-text';
    textEl.textContent = msg;

function formatDuration(seconds) {
  if (!seconds) return '0:00';
  const m = Math.floor(seconds / 60);
  const s = Math.floor(seconds % 60);
  return `${m}:${s.toString().padStart(2, '0')}`;
}

function formatDate(dateStr) {
  if (!dateStr) return '';
  return new Date(dateStr).toLocaleDateString('ru-RU', { year: 'numeric', month: 'short', day: 'numeric' });
}

function totalDuration(tracks) {
  return tracks.reduce((sum, t) => sum + (t.duration || 0), 0);
}

  function updateTrackUI(track) {
    if (!ui.title) return;
    ui.title.textContent  = track.title || 'Unknown';
    ui.artist.textContent = track.artist || '—';
    ui.cover.src          = track.cover || '';
    ui.cover.style.display = track.cover ? 'block' : 'none';

  function updatePlayBtn(playing) {
    if (!ui.playBtn) return;
    ui.playBtn.innerHTML = `<span class="material-symbols-rounded">${playing ? 'pause' : 'play_arrow'}</span>`;

  function play(track, queueTracks) {
    if (!audio) return;
    if (queueTracks) {
      state.queue = queueTracks;
      state.currentIndex = queueTracks.findIndex(t => t.id === track.id);
      if (state.currentIndex < 0) state.currentIndex = 0;
    }
    if (!state.queue.length) { state.queue = [track]; state.currentIndex = 0; }

    const current = state.queue[state.currentIndex];
    updateTrackUI(current);
    audio.src = current.audio;
    audio.play().catch(() => {});
    saveState();
  }

  function togglePlay() {
    if (!audio || !audio.src) return;
    if (audio.paused) audio.play().catch(() => {});
    else audio.pause();
  }

  function stop(clearQueue = false) {
    if (!audio) return;
    audio.pause();
    audio.currentTime = 0;
    audio.src = '';
    state.playing = false;
    state.progress = 0;
    if (clearQueue) {
      state.queue = [];
      state.currentIndex = 0;
    }
    updatePlayBtn(false);
    saveState();
  }

  function prev() {
    if (!state.queue.length) return;
    state.currentIndex = (state.currentIndex - 1 + state.queue.length) % state.queue.length;
    play(state.queue[state.currentIndex]);
  }

  function next() {
    if (!state.queue.length) return;
    if (state.repeat) { audio.currentTime = 0; audio.play().catch(() => {}); return; }
    state.currentIndex = (state.currentIndex + 1) % state.queue.length;
    play(state.queue[state.currentIndex]);
  }

  function shuffle() {
    for (let i = state.queue.length - 1; i > 0; i--) {
      const j = Math.floor(Math.random() * (i + 1));
      [state.queue[i], state.queue[j]] = [state.queue[j], state.queue[i]];
    }
    state.currentIndex = 0;
    showToast('Очередь перемешана', 'success', 1500);
    saveState();
  }

  function toggleRepeat() {
    state.repeat = !state.repeat;
    const btn = document.getElementById('player-repeat');
    if (btn) btn.style.color = state.repeat ? 'var(--rose)' : '';
    showToast(state.repeat ? 'Повтор включён' : 'Повтор выключен', 'success', 1500);
    saveState();
  }

  function requestAutoplay() {
    if (!audio) return;
    audio.play().then(() => {
      state.autoplayPending = false;
      saveState();
    }).catch(() => {
      state.autoplayPending = true;
      saveState();
    });
  }

  function bindAutoplayResume() {
    const resume = () => {
      if (!state.autoplayPending) return;
      requestAutoplay();
      if (!state.autoplayPending) {
        window.removeEventListener('click', resume, true);
        window.removeEventListener('keydown', resume, true);
      }
    };
    window.addEventListener('click', resume, true);
    window.addEventListener('keydown', resume, true);
  }

  function init() {
    initAudio();
    buildUI();

    if (state.queue.length > 0 && audio) {
      const track = state.queue[state.currentIndex || 0];
      if (track && track.audio && !audio.src.includes(track.audio)) {
        audio.src = track.audio;
        if (state.progress) audio.currentTime = state.progress;
      }
      if (ui.current && state.progress) ui.current.textContent = formatDuration(state.progress);
      if (state.playing) {
        requestAutoplay();
        bindAutoplayResume();
      }
    }
  }

  return { init, play, togglePlay, prev, next, stop };
})();
```

```js
function shuffleArray(arr) {
  const out = Array.isArray(arr) ? [...arr] : [];
  for (let i = out.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [out[i], out[j]] = [out[j], out[i]];
  }
  return out;
}
```

```js
function getAudioDuration(file) {
  log('getAudioDuration', { name: file.name, size: file.size });
  return new Promise((resolve) => {
    const audio = new Audio();
    const url = URL.createObjectURL(file);
    audio.preload = 'metadata';
    audio.src = url;
    audio.addEventListener('loadedmetadata', () => {
      const d = Number.isFinite(audio.duration) ? Math.round(audio.duration) : 0;
      URL.revokeObjectURL(url);
      resolve(d);
    });
    audio.addEventListener('error', () => {
      URL.revokeObjectURL(url);
      resolve(0);
    });
  });
}

async function ensureDurations(files) {
```

```js
(function () {
  window.__SPA__ = true;
  const logErr = (msg, err) => (window.ZError ? window.ZError('spa', msg, err) : console.error('[SPA]', msg, err || ''));

  
const log = () => {};

    '/index.html',

    '/playlist.html': 'playlist',
    '/favorite.html': 'favorite',
    '/stats.html': 'stats',
    '/archive.html': 'archive',
    '/about.html': 'about',

  };
  const routeInitFns = {
    '/index.html': 'initIndex',
    '/playlist.html': 'initPlaylist',
    '/favorite.html': 'initFavorite',
    '/stats.html': 'initStats',
    '/archive.html': 'initArchive',
    '/about.html': 'initAbout',
    '/profile.html': 'initProfile'
  };

  function normalizePath(path) {
    if (!path || path === '/') return '/index.html';
    return path;
  }

  function isSpaPath(path) {
    return spaPaths.has(normalizePath(path));
  }

  function loadScriptOnce(src) {
    return new Promise((resolve, reject) => {
      const existing = document.querySelector(`script[data-spa-src="${src}"]`);
      if (existing) {
        if (existing.dataset.spaLoaded === '1') { resolve(); return; }
        log('waiting existing script', src);
        existing.addEventListener('load', () => resolve(), { once: true });
        existing.addEventListener('error', () => reject(new Error(`Failed to load ${src}`)), { once: true });
        return;
      }
      log('inject script', src);
      const s = document.createElement('script');
      s.src = src;
      s.defer = true;
      s.dataset.spaSrc = src;
      s.onload = () => { s.dataset.spaLoaded = '1'; resolve(); };
      s.onerror = () => reject(new Error(`Failed to load ${src}`));
      document.body.appendChild(s);
    });
  }

  async function ensureRouteScript(path) {
    const src = routeScripts[normalizePath(path)];
    if (!src) return;
    await loadScriptOnce(src);
  }

  async function runRouteInit(path) {
    const key = routeInits[normalizePath(path)];
    const fnName = routeInitFns[normalizePath(path)];
    const inits = window.PageInits || {};
    const init = key ? inits[key] : null;
    log('init route', { path, key, fnName, hasPageInit: !!init });
    if (typeof init === 'function') {
      await init();
      return true;
    }
    if (fnName && typeof window[fnName] === 'function') {
      log('calling window init', fnName);
      await window[fnName]();
      return true;
    }
    return false;
  }

  async function loadPage(url, push = false) {
    const u = new URL(url, location.origin);
    const path = normalizePath(u.pathname);
    if (!isSpaPath(path)) {
      if (push) window.location.href = u.pathname + u.search;
      return;
    }

    const currentMain = document.querySelector('main');
    if (currentMain) {
      currentMain.classList.add('is-fading');
      await new Promise(r => setTimeout(r, 200));
    }

    const res = await fetch(u.pathname + u.search, { headers: { 'X-SPA': '1' } });
    log('fetch', { url: u.pathname + u.search, status: res.status });
    if (!res.ok) {
      window.location.href = u.pathname + u.search;
      return;
    }
    const html = await res.text();
    const doc = new DOMParser().parseFromString(html, 'text/html');
    const newMain = doc.querySelector('main');
    const curMain = document.querySelector('main');
    if (!newMain || !curMain) {
      window.location.href = u.pathname + u.search;
      return;
    }

    document.title = doc.title || document.title;
    curMain.replaceWith(newMain);
    newMain.classList.add('is-fading');
    requestAnimationFrame(() => newMain.classList.remove('is-fading'));

    const key = routeInits[normalizePath(path)];
    if (typeof window.renderHeader === 'function' && key) {
      window.renderHeader(key);
    }

    await ensureRouteScript(path);
    log('script ensured for', path);
    await runRouteInit(path);

    if (push) history.pushState({}, '', u.pathname + u.search);
  }

  document.addEventListener('click', (e) => {
    const a = e.target.closest('a');
    if (!a) return;
    if (a.target === '_blank' || a.hasAttribute('download') || a.dataset.noSpa) return;
    const url = new URL(a.href, location.origin);
    if (url.origin !== location.origin) return;
    const path = normalizePath(url.pathname);
    if (!isSpaPath(path)) return;
    e.preventDefault();
    log('nav click', url.pathname + url.search);
    loadPage(url.pathname + url.search, true);
  });

  window.addEventListener('popstate', () => {
    log('popstate', location.pathname + location.search);
    loadPage(location.pathname + location.search, false);
  });


  (async () => {
    const path = normalizePath(location.pathname);
    if (!isSpaPath(path)) return;
    log('bootstrap', path);
    await ensureRouteScript(path);
    await runRouteInit(path);
```

#### 2.3.25 Сценарии ошибок и валидаций
В проекте предусмотрены проверки на клиенте и сервере. Они защищают от некорректных данных и улучшают пользовательский опыт.

**Серверные проверки (пример)**
```js
app.post('/api/playlists', (req, res) => {
  const userId = req.body.userId ? parseInt(req.body.userId) : null;
  if (!userId) return res.status(403).json({ error: 'Auth required' });
  if (!req.body.title) return res.status(400).json({ error: 'Title required' });
  // дальнейшая логика
```

**Типовые ошибки, которые обрабатываются**
1. Отсутствие авторизации (`403 Auth required`).
2. Попытка редактирования чужих плейлистов (`403 Forbidden`).
3. Обращение к несуществующему треку или альбому (`404 Not found`).
4. Некорректные поля формы (`400 Bad request`).
5. Ошибка загрузки файлов через Multer (например, неверный формат).

**Поведение на клиенте**
1. Выводится toast‑сообщение об ошибке.
2. Кнопки остаются доступными, чтобы пользователь мог исправить ввод.
3. Интерфейс не «падает» и продолжает работать в текущем состоянии.

**Поведение на сервере**
1. Ошибка возвращается JSON‑ответом.
2. Данные в `data/` не изменяются, если операция не прошла.
3. Файлы в `uploads/` не создаются при ошибке загрузки.

Этот раздел завершает полное описание JavaScript‑логики, включая листинги, структуру данных и проверки.
